# V0.1 — Basic Chat CLI (TypeScript Edition)

> Talk to the model via any OpenAI-compatible provider. No tools. Just plumbing.
> This is the foundation everything else builds on.
>
> **Shipped as:** v0.0.1 (JS prototype) → v0.0.2-ts.1 (TypeScript rewrite)

---

## 1. Overview

**What it is:**
A CLI that opens a conversation with any OpenAI-compatible model provider. User types a message → model streams a response → conversation continues for multiple turns.

**What it is NOT:**

- No tool calling
- No agent loop
- No filesystem access
- No session persistence
- No fancy TUI

**Why this matters:**
This version proves the plumbing works — API call, streaming, conversation history. Every subsequent version depends on this being solid. If you can't reliably get tokens from the model, nothing else works.

## 2. Concepts to Learn

### 2.1 LLM Chat Completions API

**What it is:** The standard way to talk to a language model over HTTP. You send an array of messages (`[{role: "user", content: "..."}]`) and get back a response. The model doesn't remember past messages — you send the whole history every time.

**Why you need it:** This is the only communication channel between Beast97 and the model. Every tool call, every response, every compaction — all of it goes through this single API.

**Read:** Provider API docs → chat completions endpoint

### 2.2 OpenAI-Compatible Schema

**What it is:** A message format popularized by OpenAI that became the industry standard. Messages have `role` (system, user, assistant, tool) and `content`. The `openai` npm package wraps this into a clean API.

**Why you need it:** Every compatible provider (OpenRouter, OpenAI, Together, etc.) speaks this format. The `openai` SDK handles serialization, streaming, and error handling for you.

**Read:** OpenAI chat completions docs (format applies to any compatible provider)

### 2.3 Streaming

**What it is:** Instead of waiting for the full response, the API sends tokens one-by-one as they're generated. You get partial results immediately and render them as they arrive.

**Why you need it:** Without streaming, the user stares at a blank screen for seconds while the model thinks. With streaming, tokens appear character-by-character — the CLI feels alive.

**Read:** `openai` SDK `stream: true` option

### 2.4 Tokenization

**What it is:** Models don't read characters — they read tokens (roughly 0.75 words per token). Context windows are measured in tokens. You need to know how many tokens a message costs.

**Why you need it:** Context management (V0.7) depends on accurate token counting. For V0.1, just understand that token count ≠ character count.

**Read:** DeepSeek uses ~0.5-0.7 tokens per character for English text

### 2.5 Async/Await + Promises in Node.js

**What it is:** JavaScript's model for handling asynchronous operations (API calls, file reads, streams). `async` functions return promises, `await` pauses execution until a promise resolves.

**Why you need it:** Every API call is async. Streaming is async. Reading input is async. The entire harness runs on async/await.

**Read:** MDN async/await guide

### 2.6 TypeScript for Node.js

**What it is:** A typed superset of JavaScript. Source files (`.ts`) are compiled to JavaScript (`.js`) via `tsc`. `tsx` runs TypeScript directly during development without a build step.

**Why you need it:** TypeScript catches type errors at compile time, provides better IDE autocompletion, and documents the code's contracts (what shape is a payload? what does a function return?).

**Read:** TypeScript handbook, tsx README

### 2.7 Environment Variables & dotenv

**What it is:** Configuration values (API keys, model names) stored outside the code in a `.env` file. `dotenv` loads them into `process.env`.

**Why you need it:** You never hardcode API keys. `.env` is ignored by git (via `.gitignore`). Everyone using Beast97 creates their own `.env`.

**Read:** `dotenv` npm package README

### 2.8 CLI Input / Output

**What it is:** Reading from stdin (user keyboard) and writing to stdout (terminal display). Node.js `readline/promises` module provides a clean async interface.

**Why you need it:** The CLI reads user messages and prints model responses.

**Read:** Node.js `readline/promises` module docs

## 3. V0.1 Decisions

These are locked for V0.1. Each is documented in the ADR file.

| Decision             | Choice                                           | Rationale                                                        |
| -------------------- | ------------------------------------------------ | ---------------------------------------------------------------- |
| Provider interface   | OpenAI-compatible SDK schema                     | Any provider speaking the OpenAI format works — swap via env vars |
| Language             | TypeScript (strict)                              | Type safety, better DX, self-documenting interfaces              |
| Module system        | ESM (`import`/`export`)                          | Modern standard, openai SDK is ESM-first                         |
| OpenAI SDK           | `openai` v6.x                                    | Standard, well-documented, maintained                            |
| API key env var      | `PROVIDER_API_KEY`                               | Generic — works with any provider                                |
| Model env var        | `JUGARI_MODEL`                                   | No default — user must set this                                  |
| Base URL env var     | `PROVIDER_BASE_URL`                              | No default — user must set this                                  |
| Provider headers     | Optional, per-provider                           | OpenRouter needs HTTP-Referer + X-Title; others may differ       |
| Testing framework    | Vitest                                            | Native TS support, no experimental flags, `vi.mock` for mocking  |
| Dev runner           | `tsx`                                             | Run TypeScript directly without build step                       |
| Build tool           | `tsc`                                             | Standard TypeScript compiler to `dist/`                          |
| Package name         | `@alphabeast97/beast97`                            | Scoped npm package                                                |
| Entrypoint           | `src/index.ts` → compiled to `dist/index.js`      | Source in `src/`, output in `dist/`                               |
| CLI parsing          | `readline/promises` (built-in)                    | Node.js built-in, no deps needed                                 |
| .env loading         | `dotenv`                                          | Standard, minimal                                                 |

## 6. API Contract

The Beast97 CLI communicates with any OpenAI-compatible chat completions API. Configure the endpoint, API key, and model via environment variables.

### Endpoint

Set via `PROVIDER_BASE_URL`. Examples:

- **OpenRouter:** `https://openrouter.ai/api/v1`
- **OpenAI:** `https://api.openai.com/v1`
- **Together:** `https://api.together.xyz/v1`
- **Local (Ollama):** `http://localhost:11434/v1`

### Headers

```json
{
  "Authorization": "Bearer <PROVIDER_API_KEY>",
  "Content-Type": "application/json"
}
```

Some providers require additional headers (e.g., OpenRouter needs `HTTP-Referer` and `X-Title`). These are set in `src/llm.ts` and are not exposed to config in V0.1.

### Request Body

```json
{
  "model": "<JUGARI_MODEL>",
  "messages": [
    { "role": "system", "content": "Give one line answer..." },
    { "role": "user", "content": "Hello, what can you do?" }
  ],
  "stream": true,
  "max_tokens": 4096
}
```

### Response (streaming)

Each chunk contains a `delta` with the next token:

```
data: {"choices":[{"delta":{"content":"Hello"}}]}
data: {"choices":[{"delta":{"content":"!"}}]}
data: [DONE]
```

### Curl Example (OpenRouter)

```bash
curl https://openrouter.ai/api/v1/chat/completions \
  -H "Authorization: Bearer $PROVIDER_API_KEY" \
  -H "Content-Type: application/json" \
  -H "HTTP-Referer: https://github.com/alphaBeast97/beast97" \
  -H "X-Title: Beast97" \
  -d '{
    "model": "deepseek/deepseek-v4-flash:free",
    "messages": [{"role": "user", "content": "Say hello"}],
    "stream": false
  }'
```

### TypeScript (openai SDK)

```typescript
import OpenAI from "openai";

const openai = new OpenAI({
  apiKey: process.env.PROVIDER_API_KEY!,
  baseURL: process.env.PROVIDER_BASE_URL!,
});

const completion = await openai.chat.completions.create({
  model: process.env.JUGARI_MODEL!,
  messages: [{ role: "user", content: "Hello!" }],
  stream: true,
});

for await (const chunk of completion) {
  process.stdout.write(chunk.choices[0]?.delta?.content || "");
}
```

## 7. CLI Interface

### Usage

```bash
# Development (no build needed)
npx tsx src/index.ts

# Production (compiled)
npm run build
node dist/index.js
```

### What the user sees

```
@alphabeast97/beast97 v0.0.2-ts.1 — model: deepseek/deepseek-v4-flash:free, provider: https://openrouter.ai/api/v1
────────────────────────────────────────────────────────────────────────────────
User: 
>Hello, what can you do?
I can help you explore and understand codebases! I can read...
User: 
>exit
```

### Behavior

- **Prompt:** Shows `User: \n>` — waits for user input
- **Model response:** Streams tokens to stdout as they arrive
- **Multiline input:** Submit on Enter. No multiline editing
- **Exit commands:** Only `Ctrl+C` — graceful shutdown (typed exit intentionally omitted)
- **Errors:** Printed to stderr, don't crash
- **Empty input:** Not explicitly handled — sends empty string to the model

### Mocking strategy

Use Vitest's `vi.mock` + `vi.hoisted` to avoid hitting the real API:

```typescript
import { describe, it, expect, vi, beforeAll } from "vitest";

// Set env vars before config is loaded
process.env.PROVIDER_API_KEY = "sk-test-key";
process.env.PROVIDER_BASE_URL = "https://test-provider.com/api/v1";
process.env.JUGARI_MODEL = "test-model";

// Create mock functions before vi.mock hoisted factories run
const { mockCreate } = vi.hoisted(() => {
  async function* createStream<T>(chunks: T[]): AsyncGenerator<T> {
    for (const chunk of chunks) yield chunk;
  }

  const mockCreate = vi.fn((opts: unknown) => {
    return createStream([
      { choices: [{ delta: { content: "Hello" } }] },
      { choices: [{ delta: { content: " " } }] },
      { choices: [{ delta: { content: "World" } }] },
      { choices: [{ index: 0, delta: {} }] },
    ]);
  });

  return { mockCreate };
});

vi.mock("openai", () => ({
  default: class {
    constructor() {}
    chat = { completions: { create: mockCreate } };
  },
}));
```
