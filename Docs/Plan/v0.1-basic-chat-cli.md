# V0.1 — Basic Chat CLI

> Talk to the model via any OpenAI-compatible provider. No tools. Just plumbing.
> This is the foundation everything else builds on.

---

## 1. Overview

**What it is:**
A CLI that opens a conversation with any OpenAI-compatible model provider. User types a message → model streams a response → conversation continues for multiple turns.

**What it is NOT:**

- No tool calling
- No agent loop
- No filesystem access
- No session persistence
- No fancy TUI

**Why this matters:**
This version proves the plumbing works — API call, streaming, conversation history. Every subsequent version depends on this being solid. If you can't reliably get tokens from the model, nothing else works.

## 2. Concepts to Learn

Before writing code, understand these concepts. Each has a short explanation, why it matters for this project, and what to read.

### 2.1 LLM Chat Completions API

**What it is:** The standard way to talk to a language model over HTTP. You send an array of messages (`[{role: "user", content: "..."}]`) and get back a response. The model doesn't remember past messages — you send the whole history every time.

**Why you need it:** This is the only communication channel between Jugari and the model. Every tool call, every response, every compaction — all of it goes through this single API.

**Read:** Provider API docs → chat completions endpoint

### 2.2 OpenAI-Compatible Schema

**What it is:** A message format popularized by OpenAI that became the industry standard. Messages have `role` (system, user, assistant, tool) and `content`. The `openai` npm package wraps this into a clean JavaScript API.

**Why you need it:** Every compatible provider (OpenRouter, OpenAI, Together, etc.) speaks this format. The `openai` SDK handles serialization, streaming, and error handling for you.

**Read:** OpenAI chat completions docs (format applies to any compatible provider)

### 2.3 Streaming

**What it is:** Instead of waiting for the full response, the API sends tokens one-by-one as they're generated. You get partial results immediately and render them as they arrive.

**Why you need it:** Without streaming, the user stares at a blank screen for seconds while the model thinks. With streaming, tokens appear character-by-character — the CLI feels alive.

**Read:** `openai` SDK `stream: true` option

### 2.4 Tokenization

**What it is:** Models don't read characters — they read tokens (roughly 0.75 words per token). Context windows are measured in tokens. You need to know how many tokens a message costs.

**Why you need it:** Context management (V0.7) depends on accurate token counting. For V0.1, just understand that token count ≠ character count.

**Read:** DeepSeek uses ~0.5-0.7 tokens per character for English text

### 2.5 Async/Await + Promises in Node.js

**What it is:** JavaScript's model for handling asynchronous operations (API calls, file reads, streams). `async` functions return promises, `await` pauses execution until a promise resolves.

**Why you need it:** Every API call is async. Streaming is async. Reading input is async. The entire harness runs on async/await.

**Read:** MDN async/await guide

### 2.6 Node.js Streams

**What it is:** A stream is a sequence of data chunks arriving over time. You can process each chunk as it arrives instead of waiting for the entire payload.

**Why you need it:** LLM streaming responses are Node.js ReadableStreams. You'll iterate over chunks and print tokens as they arrive.

**Read:** Node.js stream docs, `for await...of` with async iterables

### 2.7 Environment Variables & dotenv

**What it is:** Configuration values (API keys, model names) stored outside the code in a `.env` file. `dotenv` loads them into `process.env`.

**Why you need it:** You never hardcode API keys. `.env` is ignored by git (via `.gitignore`). Everyone using Jugari creates their own `.env`.

**Read:** `dotenv` npm package README

### 2.8 CLI Input / Output

**What it is:** Reading from stdin (user keyboard) and writing to stdout (terminal display). Raw Node.js `process.stdin` and `process.stdout`.

**Why you need it:** The CLI reads user messages and prints model responses. Simple now — gets more complex with readline, prompts, and eventual TUI.

**Read:** Node.js `readline` module (for V0.1, just `process.stdin`)

## 3. V0.1 Decisions

These are locked for V0.1. Each is documented in the ADR file.

| Decision           | Choice                       | Rationale                                                         |
| ------------------ | ---------------------------- | ----------------------------------------------------------------- |
| Provider interface | OpenAI-compatible SDK schema | Any provider speaking the OpenAI format works — swap via env vars |
| Module system      | ESM (`import`/`export`)      | Modern standard, openai SDK is ESM-first                          |
| OpenAI SDK         | `openai` v6.x                | Standard, well-documented, maintained                             |
| API key env var    | `PROVIDER_API_KEY`           | Generic — works with any provider                                 |
| Model env var      | `JUGARI_MODEL`               | No default — user must set this                                   |
| Base URL env var   | `PROVIDER_BASE_URL`          | No default — user must set this                                   |
| Provider headers   | Optional, per-provider       | OpenRouter needs HTTP-Referer + X-Title; others may differ        |
| Testing framework  | Node built-in `node:test`    | Zero deps, ESM-native, available in Node 18+                      |
| Package name       | `@alphabeast97/jugari`       | Scoped, with `"private": true`                                    |
| Entrypoint         | `src/index.js`               | Standard Node convention                                          |
| CLI parsing        | Raw `process.argv`           | Simplest possible — no deps for V0.1                              |
| .env loading       | `dotenv`                     | Standard, minimal                                                 |

## 6. API Contract

The Jugari CLI communicates with any OpenAI-compatible chat completions API. Configure the endpoint, API key, and model via environment variables.

### Endpoint

Set via `PROVIDER_BASE_URL`. Examples:

- **OpenRouter:** `https://openrouter.ai/api/v1`
- **OpenAI:** `https://api.openai.com/v1`
- **Together:** `https://api.together.xyz/v1`
- **Local (Ollama):** `http://localhost:11434/v1`

### Headers

```json
{
  "Authorization": "Bearer <PROVIDER_API_KEY>",
  "Content-Type": "application/json"
}
```

Some providers require additional headers (e.g., OpenRouter needs `HTTP-Referer` and `X-Title`). These are set in `src/llm.js` and are not exposed to config in V0.1.

### Request Body

```json
{
  "model": "<JUGARI_MODEL>",
  "messages": [
    { "role": "user", "content": "Hello, what can you do?" }
  ],
  "stream": true,
  "max_tokens": 4096
}
```

### Response (streaming)

Each chunk contains a `delta` with the next token:

```
data: {"choices":[{"delta":{"content":"Hello"}}]}
data: {"choices":[{"delta":{"content":"!"}}]}
data: [DONE]
```

### Curl Example (OpenRouter)

```bash
curl https://openrouter.ai/api/v1/chat/completions \
  -H "Authorization: Bearer $PROVIDER_API_KEY" \
  -H "Content-Type: application/json" \
  -H "HTTP-Referer: https://github.com/AlphaBeast97/Jugari" \
  -H "X-Title: Jugari" \
  -d '{
    "model": "deepseek/deepseek-v4-flash:free",
    "messages": [{"role": "user", "content": "Say hello"}],
    "stream": false
  }'
```

### Node.js (openai SDK)

```javascript
import OpenAI from 'openai';
import 'dotenv/config';

const client = new OpenAI({
  apiKey: process.env.PROVIDER_API_KEY,
  baseURL: process.env.PROVIDER_BASE_URL,
});

const stream = await client.chat.completions.create({
  model: process.env.JUGARI_MODEL,
  messages: [{ role: 'user', content: 'Hello!' }],
  stream: true,
});

for await (const chunk of stream) {
  process.stdout.write(chunk.choices[0]?.delta?.content || '');
}
```

## 7. CLI Interface

### Usage

```bash
node src/index.js
```

### What the user sees

```
Jugari v0.1 — provider: <PROVIDER_BASE_URL>, model: <JUGARI_MODEL>
────────────────────────────────────────────────────────────────
> Hello, what can you do?
< I can help you explore and understand codebases! I can read...
> exit
Goodbye.
```

### Behavior

- **Prompt:** Shows `> ` prefix, waits for user input
- **Model response:** Streams tokens to stdout with `< ` prefix
- **Multiline input:** Submit on Enter. No multiline editing (V0.1)
- **Exit commands:** `exit`, `quit`, `Ctrl+C` — graceful shutdown
- **Errors:** Red text to stderr, don't crash, return to prompt
- **Empty input:** Ignore silently, show prompt again

## 10. Tests Reference

### Running tests

### Test file structure

```
tests/
├── llm.test.js          # Mock API responses, test parsing
├── history.test.js      # Test array operations

```

### Mocking strategy

Use `node:test` mocks (available in Node 18+) to avoid hitting the real API in tests:

```javascript
import { describe, it, mock } from 'node:test';
import assert from 'node:assert';

// Mock the OpenAI client for llm tests
const mockCreate = mock.fn();
mock.module('openai', {
  defaultExport: class {
    constructor() {}
    chat = { completions: { create: mockCreate } };
  },
});
```
